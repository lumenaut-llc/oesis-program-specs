# Data Ownership

## Purpose

State the homeowner-facing data ownership and control model for the April 14, 2026
open-release period.

This file is a project policy draft, not legal advice.

## Core rule

Private by default.
Shared by choice.
Homeowners should own the raw data generated by their parcel deployments.

## What the homeowner owns

The project’s working rule is that the homeowner owns and controls raw parcel-linked observations generated by sensors or devices they deploy on their property or for their household use.

This includes, where applicable:

- raw sensor observations
- parcel-linked device data
- exact parcel-linked histories associated with that homeowner account
- parcel metadata or notes supplied by the homeowner
- house-state telemetry and indoor environmental response data
- capability, compatibility, and control-locality records for parcel systems
- intervention and verification histories tied to that parcel

## What the platform receives

To operate the product, the platform may need a limited right to:

- receive homeowner-contributed data
- store it for product operation
- normalize and process it
- generate parcel-state outputs for the homeowner
- apply the sharing settings the homeowner selected
- retain limited governance and security records

The platform does not receive a blanket ownership transfer merely because data is uploaded or processed.

## Private, shared, and public are different

The project draws a hard distinction among:

- private parcel data
- shared opt-in contributions
- public context from third-party or public sources
- derived parcel-state outputs

Open-source release of code or documentation does not convert future
participant-contributed private homeowner data into public data.

The project-controlled v1 dataset may still be intentionally public when it is
explicitly designated and released under dataset-specific terms.

## Sharing rule

No parcel-linked data should leave the private parcel context unless the applicable sharing mode allows it.

Current release rule:

- private use is the default
- network assist is separate opt-in
- neighborhood aggregate contribution is separate opt-in
- research or pilot contribution requires separate, explicit opt-in

These sharing modes govern data use and visibility.
They do not authorize parcel control or automation.

See `permissions-matrix.md`.

## What the platform may do

Subject to the selected sharing mode and later product terms, the platform may:

- use private parcel data to provide the homeowner’s parcel view and related product operation
- use selected opt-in signals for internal service operation where the user has enabled that mode
- use public context to interpret parcel conditions
- generate derived parcel-state outputs, confidence values, freshness values, and provenance summaries
- intentionally publish the project-controlled v1 dataset or derived public release artifacts only where the project has clear authority and explicit dataset terms are attached

## What the platform may not do under the current policy direction

The platform should not:

- treat private parcel-linked data as ordinary marketing or advertising telemetry
- sell or disclose exact parcel-linked raw data by default
- expose exact parcel-linked contributions in public maps
- represent opt-in shared data as anonymous unless a real standard supports that claim
- silently repurpose private parcel data into broader sharing modes
- infer that device compatibility or control capability implies homeowner approval to actuate it

## Derived outputs

Derived parcel-state outputs are platform-generated interpretations about the homeowner’s parcel.

Those outputs do not erase the homeowner’s rights in the underlying raw data.

The project’s current policy direction is:

- the homeowner should be able to view current parcel-state outputs tied to their parcel
- the platform may retain derived outputs for product operation and recent history features
- derived outputs should remain labeled as estimates or inferences, not guaranteed facts

## Export, deletion, and revocation

The working rule is that homeowner ownership should be supported by practical controls, not branding language alone.

The project therefore intends to support:

- export of homeowner-contributed observations and key parcel-state history
- deletion of private parcel-linked data subject to stated exceptions
- revocation of future sharing under opt-in modes

See `retention-export-deletion-revocation.md`.

## Release limitations

This file states the intended policy posture. It does not guarantee that every control
is fully implemented today.

Before any pilot or broader public deployment, the project should confirm:

- what controls already work in software
- what remains roadmap-only
- what exceptions apply for security, abuse prevention, or legal obligations
