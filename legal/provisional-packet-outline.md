# Provisional Packet Outline

## Purpose

Use this outline to prepare a narrow U.S. provisional filing packet before the enabling open release.

This file is planning support only. Patent filing strategy requires patent-attorney review if available.

## Packet objective

Capture one realistically fileable narrow invention before the team makes enabling public disclosures.

Do not try to cover the whole project. Choose the smallest combination that would still matter if copied by others.

## Recommended candidate types

Pick one primary target:

- parcel-level status generation using parcel priors plus local, shared, and public evidence with explicit evidence mode
- privacy-scoped neighborhood contribution method for shared intelligence without exposing parcel-linked raw data
- a specific hardware-to-platform-to-parcel inference interaction

## Packet metadata

- working title:
- inventors:
- owner/applicant to confirm with counsel:
- drafting owner:
- target filing date:
- related public materials to hold until filing:

## Section 1. Problem statement

Capture:

- what practical problem exists in current regional, centralized, or coarse-grained systems
- why parcel-level interpretation matters
- why parcel operator control and partial-adoption utility matter
- what specific technical gap the narrow invention addresses

## Section 2. Narrow invention summary

Capture:

- one paragraph that states the invention candidate precisely
- what inputs it uses
- what transformation or method it performs
- what outputs it produces
- why this combination is different from ordinary sensor dashboards or regional hazard feeds

## Section 3. System context

Capture:

- the broader system modules the invention sits inside
- where the invention begins and ends
- what pieces are conventional background versus claimed candidate material

Keep this section contextual. Do not let it sprawl into an attempt to patent the whole platform.

## Section 4. Definitions

Define terms that matter to the candidate invention, for example:

- parcel prior
- local evidence
- shared evidence
- public context
- evidence mode
- confidence
- derived parcel state
- neighborhood contribution

## Section 5. Detailed method description

Describe the candidate in enough detail that it is supported later.

Include:

- ordered steps
- data inputs and outputs
- intermediate transformations
- alternative embodiments
- fallback behaviors
- examples for smoke, flooding/runoff, and heat if relevant

If a detail feels important enough that you would regret losing it, include it here.

## Section 6. Figures and diagrams

Prepare labeled figures for:

- system block diagram
- method flowchart
- data transformation flow
- user permission boundary diagram
- example parcel-state generation flow
- example neighborhood contribution flow

Each figure should have:

- figure number
- short title
- one-paragraph explanation

## Section 7. Embodiments and variants

List variants such as:

- different node types
- local-first versus cloud-assisted processing
- different confidence treatments
- different neighborhood granularity choices
- event-triggered sharing versus continuous contribution

If a variant may matter later, describe it now.

## Section 8. Example implementations

Provide concrete worked examples showing:

- input conditions
- processing steps
- output states
- evidence mode
- confidence behavior

Use synthetic examples only.

## Section 9. Distinguishing points

State what you believe is distinctive compared with:

- ordinary environmental monitoring dashboards
- public regional alerts
- generic neighborhood maps
- simple private sensor logging

This section helps counsel focus quickly, even if the wording is later refined.

## Section 10. Known alternatives and prior art notes

Capture:

- known adjacent systems
- known public datasets or approaches
- what is similar
- what you think is different

Do not over-polish this. A rough honest list is more useful than silence.

## Section 11. Filing dependencies

Before filing, confirm:

- inventor list is reviewed
- the narrow invention is chosen
- figures are attached
- the packet covers the details you want a filing date for
- public-preview materials do not exceed what the packet supports

## Section 12. Post-filing release rule

After filing:

- release only material that is supported by the filed packet, or
- deliberately treat new material as a separate open publication decision, or
- prepare a follow-on filing before publishing the new material

## Quick-start fields to fill first

If time is tight, fill these first:

- working title
- inventors
- one-sentence narrow invention summary
- problem statement
- figure list
- top three variants to preserve
