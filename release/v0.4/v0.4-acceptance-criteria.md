# v0.4 Acceptance Criteria

## Purpose

Define concrete, testable acceptance criteria that must pass before promoting
from v0.3 to v0.4.

## Acceptance tests

### AC-1: Node registration lifecycle

**Given** a parcel with `parcel_id: "parcel_test_v04"`

**When** the following lifecycle is exercised:
1. Register `bench-air-01` (indoor) → bind to parcel → enabled
2. Register `mast-lite-01` (outdoor) → bind to parcel → enabled
3. Disable `mast-lite-01` (maintenance)
4. Register `mast-lite-02` (replacement) → bind to parcel → enabled
5. Retire `mast-lite-01`

**Then** the registry reflects: `bench-air-01` enabled, `mast-lite-02` enabled,
`mast-lite-01` retired. Parcel-state inference uses only enabled nodes.

### AC-2: Installation metadata capture

**Given** a newly registered node `bench-air-02`

**When** an operator submits deployment metadata (mount type, height,
orientation, exposure, power source)

**Then** the metadata is stored as a valid `deployment-metadata` record linked
to the node's `install_record_id`, and can be retrieved via API.

### AC-3: Evidence composition with quality weighting

**Given** parcel `parcel_test_v04` has:
- a fresh indoor observation from a field-validated bench-air node
- a stale sheltered outdoor observation from a provisional mast-lite node
- a fresh public context source

**When** `GET /v1/parcels/parcel_test_v04/state` is called

**Then** the parcel-state:
- weights the fresh indoor observation higher than the stale outdoor
- reflects deployment grade in provenance (field_validated vs provisional)
- explanation includes weighting rationale

### AC-4: Deployment-quality flags visible

**Given** a parcel with one bench-grade node and one field-validated node

**When** the parcel view is rendered

**Then** the view surfaces the deployment grade distinction for each
contributing node, and the evidence summary reflects overall deployment posture.

### AC-5: Node replacement continuity

**Given** parcel `parcel_test_v04` with historical observations from
`mast-lite-01` (retired) and current observations from `mast-lite-02` (active)

**When** parcel-state is computed

**Then** inference uses only `mast-lite-02` observations for current state;
historical observations from `mast-lite-01` remain accessible but do not
contribute to current inference.

### AC-6: v0.3 regression

**Given** the v0.4 codebase

**When** all prior acceptance suites run (`make oesis-validate`,
`make oesis-v02-accept`, `make oesis-v03-accept`)

**Then** all prior acceptance tests continue to pass.

## Contract boundary changes (v0.3 → v0.4)

- Node registry adds lifecycle state field (active, disabled, retired)
- Node registry adds deployment_grade field (bench_grade, field_ready)
- Deployment-metadata contract integrated as input to evidence composition
- Parcel-state provenance includes source weighting rationale
- No breaking changes to existing schemas

## Acceptance command (proposed)

```
make oesis-v04-accept
```
