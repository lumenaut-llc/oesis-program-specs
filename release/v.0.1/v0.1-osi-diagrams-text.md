## 1. Bench-air serial path (hardware trace → operator host)

USB CDC serial from ESP32-S3 to the operator machine. **I2C from sensors to the MCU is on-board signaling, not OSI.** OSI applies to the **USB byte stream**.

### Hardware inventory (v0.1 bench-air kit)

| Item | Role |
| --- | --- |
| SHT45 breakout | Temperature and RH (primary); I2C to ESP32 |
| BME680 breakout | Pressure, gas-resistance trend, secondary RH/temp; shared I2C bus |
| ESP32-S3 DevKitC-1 | Firmware, JSON line emission; GPIO8 SDA, GPIO9 SCL per build guide |
| USB cable (data-capable) | USB CDC serial to operator PC |
| Operator PC | Serial monitor, capture, `extract_latest_packet`, `ingest_packet` |

### Hardware connectivity (ASCII)

```
   +----------+   +----------+
   |  SHT45   |   | BME680   |
   +----+-----+   +----+-----+
        | I2C          | I2C
        |    shared bus (3V3, GND, SDA, SCL)
        v              v
   +---------------------------+
   | ESP32-S3 DevKitC-1        |
   | firmware: bench-air JSON  |
   +-------------+-------------+
                 | USB CDC
                 v
   +---------------------------+
   | USB cable                 |
   +-------------+-------------+
                 |
                 v
   +---------------------------+
   | Operator PC               |
   +---------------------------+
```

### OSI table (USB serial leg)

| Layer | Name | v0.1 role |
| --- | --- | --- |
| L7 | Application | Firmware emits JSON lines; host tools parse `oesis.bench-air.v1` |
| L6 | Presentation | UTF-8 text, JSON syntax |
| L5 | Session | None; opaque stream of lines |
| L4 | Transport | Byte stream over USB CDC or UART |
| L3 | Network | Not used; point-to-point attachment |
| L2 | Data link | USB CDC framing or UART characters |
| L1 | Physical | USB D+/D-, plugs, ESP32 and host USB PHY |

### OSI ASCII stack

```
+------------------------------------------------------------------+
| L7  Application   | JSON lines; oesis tools on host             |
+------------------------------------------------------------------+
| L6  Presentation  | UTF-8 JSON                                   |
+------------------------------------------------------------------+
| L5  Session       | none; stream of lines                        |
+------------------------------------------------------------------+
| L4  Transport     | byte stream over USB CDC                     |
+------------------------------------------------------------------+
| L3  Network       | not used                                     |
+------------------------------------------------------------------+
| L2  Data link     | USB CDC / UART framing                       |
+------------------------------------------------------------------+
| L1  Physical      | USB PHY both ends, cable                     |
+------------------------------------------------------------------+
```

## 2. Offline reference pipeline and acceptance (no network)

`make oesis-validate`, `make oesis-check`, `make oesis-accept`: examples on disk; **one machine**. Real bench hardware is **off-path**; JSON fixtures represent captured packets.

### Hardware (single host)

| Item | Role |
| --- | --- |
| Operator workstation | CPU, RAM |
| Local disk | Packaged `oesis/assets/examples/*.json` |

         ^

+------------------+
```

| Layer | Name | v0.1 role |
| --- | --- | --- |
| L7 | Application | Python `oesis` modules; JSON fixtures on disk |

| L7  Application   | Python oesis modules, JSON fixtures on disk  |
+------------------------------------------------------------------+
| L6–L1             | not applicable: single process, no wire      |
+------------------------------------------------------------------+
```

## 3. Local HTTP reference (`make oesis-http-check`)

Three services on **loopback** on **one PC**. No USB node required for the smoke check.

### Hardware

| Item | Role |
| --- | --- |
| Same operator PC | Three Python HTTP listeners |
| Virtual loopback NIC | `127.0.0.1`; no external Ethernet/Wi-Fi needed |

### ASCII

```
+---------------------------+
| Operator PC               |
|  lo0 / loopback           |
|  :8787 ingest             |
|  :8788 inference          |
|  :8789 parcel-platform    |
+---------------------------+
```

### OSI table

| Layer | Name | v0.1 role |
| --- | --- | --- |
| L7 | Application | HTTP JSON REST shapes; `X-OESIS-Parcel-Id` header |
| L6 | Presentation | JSON bodies, UTF-8 |
| L5 | Session | HTTP keep-alive optional; per-request model |
| L4 | Transport | TCP; ports 8787–8789 typical |
| L3 | Network | IPv4 loopback `127.0.0.1` |
| L2 | Data link | Loopback adapter (logical L2) |
| L1 | Physical | OS loopback; no external PHY |

### OSI ASCII stack

```
+------------------------------------------------------------------+
| L7  Application   | HTTP + JSON; X-OESIS-Parcel-Id               |
+------------------------------------------------------------------+
| ... through L1    | loopback stack (no external hardware)       |
+------------------------------------------------------------------+
   ingest API ---- inference API ---- parcel-platform API
```

## 4. Deferred path: node Wi-Fi to ingest (gap register G3)

### Hardware (target deployment)

| Item | Role |
| --- | --- |
| Node | Same sensor + ESP32-S3 kit; **Wi-Fi radio** used (not only USB) |
| AP or home router | 802.11 access; may bridge to Ethernet |
| Ingest host | Machine running ingest API (could be same PC or another host) |

### ASCII

```
 [Sensors]--I2C--[ ESP32-S3 Wi-Fi ]
                        |
                   802.11 WLAN
                        v
                 [ AP / router ]
                        |
                 Ethernet or Wi-Fi
                        v
                 [ Ingest server ]
```

### OSI table

| Layer | Name | Expected future role |
| --- | --- | --- |
| L7 | Application | Ingest API; authentication; parcel binding |
| L6 | Presentation | TLS ciphertext on wire when enabled |
| L5 | Session | TLS session or stateless HTTP |
| L4 | Transport | TCP to ingest listener |
| L3 | Network | IPv4 or IPv6 on WLAN |
| L2 | Data link | Wi-Fi MAC (802.11) |
| L1 | Physical | RF channel and antenna at node and AP |

### OSI ASCII stack

```
+------------------------------------------------------------------+
| L7  Application   | ingest API, auth, parcel binding             |
+------------------------------------------------------------------+
| ...                                                               |
+------------------------------------------------------------------+
| L1  Physical      | RF / antenna (node + AP)                     |
+------------------------------------------------------------------+
```

## 5. Deferred path: live public weather and smoke context (gap register G4)

### Hardware (WAN path)

| Item | Role |
| --- | --- |
| Runtime host | PC or server running inference with live adapters |
| CPE | Customer modem/router |
| ISP plant | Fiber, coax, or other PHY to provider |
| Remote API | Provider datacenter (logical endpoint) |

### ASCII

```
 [Runtime host]--Ethernet or Wi-Fi--[ CPE ]--ISP--[ Provider API ]
```

### OSI table

| Layer | Name | Expected future role |
| --- | --- | --- |
| L7 | Application | Provider API (JSON or XML); inference adapter |
| L6 | Presentation | TLS records; negotiated content types |
| L5 | Session | TLS 1.2 or 1.3 |
| L4 | Transport | TCP to provider |
| L3 | Network | Internet routing and DNS |
| L2 | Data link | Ethernet or Wi-Fi MAC to CPE |
| L1 | Physical | PHY toward ISP |

### OSI ASCII stack

```
+------------------------------------------------------------------+
| L7 ... L1         | as table; physical NIC on host to CPE to WAN |
+------------------------------------------------------------------+
```

## 7. Pilot Tier A vs Tier B (trust boundary)

### Hardware by tier

| Tier | Typical hardware |
| --- | --- |
| A | Bench-air node + USB + **one** workstation (figures 1–3) |
| B | Adds Wi-Fi or WAN paths, CPE, ISP, and often **separate** or **hosted** ingest or parcel services (figures 4–5) |

### ASCII (tiers)

```
 Tier A                          Tier B (adds)
+---------------------+          +-------------------------+
| Bench node          |          | Wi-Fi / WAN radios      |
| USB to one PC       |   --->   | CPE, ISP                |
| No participant PII  |          | Hosted ingest optional  |
+---------------------+          +-------------------------+
```

### Relationship

```
  Tier_A  --adds requirements-->  Tier_B
```
