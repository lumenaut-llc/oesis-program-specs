# v0.1 OSI layer reference diagrams

## Purpose

Use the **OSI model** as a consistent lens on **where** each v0.1 reference path sits in a communication stack. Figures pair **hardware** (sensors, boards, cables, PCs, radios) with **OSI layers** so reviewers see both the physical kit and the protocol stack. Layers **5–6** are included when they clarify TLS, HTTP, or encoding.

**Plain-text and ASCII versions (no Mermaid):** `v0.1-osi-diagrams-text.md`.

These figures pair with:

- `v0.1-scope-matrix.md` — what the slice claims
- `v0.1-gap-register.md` — deferred stacks (Wi-Fi ingest, live public feeds)
- `v0.1-pilot-minimum-subset.md` — internal vs external pilot boundaries
- `implementation-status-matrix.md` — surface maturity

## 1. Bench-air serial path (hardware trace → operator host)

USB CDC serial from ESP32-S3 to the operator machine: no IP, no TCP. OESIS payload is **application-layer** JSON carried as UTF-8 lines. **Sensing is not an OSI layer**; I2C from sensors to the MCU is on-board electrical signaling. OSI applies to the **USB serial byte stream** between the dev kit and the host.

### Hardware block (bench-air v0.1 kit)

```mermaid
flowchart LR
  subgraph benchAir [Bench_air_node]
    sht[SHT45_breakout]
    bme[BME680_breakout]
    esp[ESP32_S3_DevKitC_1]
    sht -->|I2C_GPIO8_SDA_GPIO9_SCL| esp
    bme -->|I2C_shared_bus| esp
  end
  usb[USB_cable_CDC_serial]
  pc[Operator_PC]
  esp --> usb --> pc
```

### OSI stack (USB serial leg, host parses JSON at L7)

```mermaid
flowchart TB
  subgraph pathSerial [OSI_USB_serial_path]
    direction TB
    L7s["L7 Application: firmware emits JSON lines; host tools parse oesis.bench-air.v1"]
    L6s["L6 Presentation: UTF-8 text, JSON syntax"]
    L5s["L5 Session: none; opaque stream of lines"]
    L4s["L4 Transport: byte stream over USB CDC or UART"]
    L3s["L3 Network: not used; point-to-point attachment"]
    L2s["L2 Data link: USB CDC framing or UART characters"]
    L1s["L1 Physical: USB D+/D-, plugs, ESP32 and host USB PHY"]
  end
```

## 2. Offline reference pipeline and acceptance (no network)

`make oesis-validate`, `make oesis-check`, `make oesis-accept`: examples on disk; normalization and inference **in-process**. **No on-wire protocol**: OSI L7 is software and file I/O on one machine. Real nodes are **off-path**; fixtures stand in for captured packets.

### Hardware (single host only)

```mermaid
flowchart TB
  subgraph oneHost [Operator_workstation]
    disk[SSD_or_NVMe_JSON_examples]
    cpu[CPU_RAM_Python_oesis_runtime]
    disk --> cpu
  end
```

### OSI (logical)

```mermaid
flowchart TB
  subgraph pathOffline [Offline_CLI_reference]
    direction TB
    L7o["L7 Application: Python oesis modules, JSON fixtures on disk"]
    L1o["L1 through L6: not applicable; single OS process, no network"]
  end
```

## 3. Local HTTP reference (`make oesis-http-check`)

Three services on **loopback**: ingest, inference, parcel-platform. All traffic stays on **one physical machine**; no bench node required for this check (fixtures drive requests).

### Hardware (loopback on one PC)

```mermaid
flowchart TB
  subgraph workstation [Same_operator_PC_as_fig_2]
    nic[Virtual_loopback_NIC]
    cpu[CPU_processes_three_HTTP_servers]
    nic <--> cpu
  end
```

### OSI stack

```mermaid
flowchart TB
  subgraph pathHttp [Localhost_HTTP_v0_1]
    direction TB
    L7h["L7 Application: HTTP JSON REST shapes, X-OESIS-Parcel-Id header"]
    L6h["L6 Presentation: JSON bodies, UTF-8"]
    L5h["L5 Session: HTTP keep-alive optional; per-request model"]
    L4h["L4 Transport: TCP ports 8787-8789 typical"]
    L3h["L3 Network: IPv4 loopback 127.0.0.1"]
    L2h["L2 Data link: loopback adapter logical L2"]
    L1h["L1 Physical: OS loopback; no external PHY"]
  end
```

## 4. Deferred path: node Wi-Fi to ingest (gap register G3)

Not required for frozen v0.1. **Hardware** adds the ESP32 **Wi-Fi radio**, an access point or home router, and a machine running the ingest service.

### Hardware (target deployment)

```mermaid
flowchart LR
  subgraph node [Bench_or_field_node]
    sens[Sensors_I2C]
    esp[ESP32_S3_Wi_Fi]
    sens --> esp
  end
  ap[Wi_Fi_AP_or_router]
  srv[Ingest_server_host]
  node -->|80211_WLAN| ap -->|Ethernet_or_WiFi| srv
```

### OSI stack (WLAN and onward)

```mermaid
flowchart TB
  subgraph pathWifi [Deferred_WiFi_to_ingest]
    direction TB
    L7w["L7 Application: ingest API, authentication, parcel binding"]
    L6w["L6 Presentation: TLS ciphertext on wire when enabled"]
    L5w["L5 Session: TLS session or stateless HTTP"]
    L4w["L4 Transport: TCP to ingest listener"]
    L3w["L3 Network: IPv4 or IPv6 on WLAN"]
    L2w["L2 Data link: Wi-Fi MAC 802.11"]
    L1w["L1 Physical: RF channel, antenna at node and AP"]
  end
```

## 5. Deferred path: live public weather and smoke context (gap register G4)

Operator-configured **HTTPS** to internet providers. **Hardware** is the operator or server **NIC**, CPE (modem/router), and ISP physical plant.

### Hardware (WAN path)

```mermaid
flowchart LR
  host[Runtime_host_PC_or_server]
  cpe[Home_or_datacenter_CPE]
  isp[ISP_physical_plant]
  cloud[Provider_API_endpoints]
  host -->|Ethernet_or_Wi_Fi| cpe --> isp --> cloud
```

### OSI stack

```mermaid
flowchart TB
  subgraph pathPublic [Deferred_public_context_feeds]
    direction TB
    L7p["L7 Application: provider API JSON or XML; inference adapter"]
    L6p["L6 Presentation: TLS records; negotiated content types"]
    L5p["L5 Session: TLS 1.2 or 1.3"]
    L4p["L4 Transport: TCP to provider"]
    L3p["L3 Network: internet routing and DNS"]
    L2p["L2 Data link: Ethernet or Wi-Fi MAC to CPE"]
    L1p["L1 Physical: PHY to ISP path"]
  end
```
