# v0.1 vs PRD scope matrix

## Purpose

Separate the **frozen v0.1 reference slice** (one parcel, one bench-air node, one ingest-to-parcel-view path) from **current v1** product requirements in `architecture/system/product-requirements-phase-1.md`, so “missing for v0.1” is not confused with deferred product work.

## Sign-off sentence (working definition)

**v0.1** means: a repeatable **reference** path from `oesis.bench-air.v1` observations through ingest, inference, and parcel view (CLI and local HTTP APIs), with checked-in examples and acceptance commands; it does **not** mean a full parcel app, push alerting, continuous Wi-Fi transport to ingest, named retail BOMs, or multi-hazard hardware beyond what bench-air plus fixtures can support.

| PRD area | In v0.1 reference slice | Notes |
| --- | --- | --- |
| **Core jobs** (what / why / basis / actions / prepare) | Partial | Reference `parcel_view` + `evidence_summary` support “what” and “basis” in JSON; no consumer UX for the rest. |
| **Readiness cards** (six states + label, confidence, freshness, why, steps) | Partial | Parcel-state / parcel-view fields map to shelter, reentry, egress, asset risk, etc.; not a card UI. Air/heat/flood/power/route breadth is limited by single-node + fixture public context. |
| **Event timeline** | Out | No persisted timeline product surface in v0.1. |
| **Evidence view** | Partial | `evidence_summary` and provenance in reference output; not a dedicated UI. |
| **FR1** parcel-state generation | In | Offline + HTTP reference path with local obs, parcel context, optional public context (fixtures). |
| **FR2** recommendation engine | Partial | Reason strings / next-step flavor in reference inference; not a separate policy engine or parcel operator-tuned copy layer. |
| **FR3** trend detection | Partial / defer | Depends on history; reference path is snapshot-oriented. Treat full FR3 as post–v0.1 unless scope expands. |
| **FR4** alerting | Out | No notifications in v0.1 reference stack. |
| **FR5** sensor-health awareness | Partial | Node `health` and normalization carry signals; parcel view exposes limits in reference form, not a health UI. |
| **FR6** setup / context capture | Partial | `parcel_context` fixture + `location_mode` on node; no guided setup product. |
| **FR7** honest stage boundary | In (discipline) | Docs and reference outputs must not claim HVAC verification, automation, etc. |
| **NFR1** truthfulness | In | Low-confidence / unknown paths in inference config and checks. |
| **NFR2** privacy | Partial | Private-by-default in architecture; sharing/revocation product guarantees are not v0.1-complete (see implementation status matrix). |
| **NFR3** explainability | Partial | Reference JSON explanations; not end-user copy. |
| **NFR4** graceful degradation | Partial | Demonstrated with fixtures; production public-context failure modes not fully specified for v0.1. |
| **NFR5** low operational burden | N/A reference | Operator quickstart exists; parcel operator burden is out of scope for v0.1 slice. |
| **NFR6** democratic / operator-controlled | Partial | Direction in docs; operational governance surfaces largely post–v0.1. |
| **In-scope hazards (PRD list)** | Partial | Strongest: air / shelter-adjacent signals from bench-air + smoke context fixtures. Flood/heat/freeze/outage readiness labels may appear with weak evidence; avoid overclaim vs sensors. |

## Hazard coverage honesty (v0.1)

| Hazard | v0.1 honesty bar |
| --- | --- |
| Smoke / indoor air burden | Primary fit for bench-air + public smoke fixture path. |
| Heat burden | Partially supported via temperature-related signals + weather fixture; not whole-home thermal mapping. |
| Flood / runoff | **Defer** unless / until low-point sensing exists (`flood-node` is planned, not reference). |
| Freeze / cold | Partial via temperature signals; no dedicated freeze instrumentation in v0.1 kit. |
| Outage / shelter | Inference/readiness may reference shelter/egress style states; no backup-power sensing in v0.1 kit. |
