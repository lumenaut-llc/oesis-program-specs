# v0.1 internal reference vs external pilot — launch checklist subset

## Purpose

Map `launch-readiness-checklist.md` to a **minimum subset** for two tiers:

1. **Internal reference v0.1** — engineers repeat hardware + runtime acceptance; no external participants.
2. **External pilot** — any real household or participant data; stop-ship rules apply.

## Tier A: Internal reference v0.1 (minimum)

Enough to align with the frozen slice and April 2026 matrix posture.

| Launch checklist section | Minimum items | Rationale |
| --- | --- | --- |
| Governance and legal | Data classes fixed for reference path (`make oesis-validate` + services use examples) | Already in progress per checklist; keep schemas stable. |
| Product and UX | Provenance and freshness visible in reference `parcel_view` | Marked done in checklist; maintain on pipeline changes. |
| Data and API | None beyond reference parcel API behavior | Sharing/consent/revocation full implementation not required. |
| Security and operator | Production-like demos avoid real household data | Marked done; keep using fixtures for demos. |
| Pilot | **None required** | No external participants. |
| Release copy | Public copy avoids directive safety language; stays within publication limits | If anything is published, gate through existing release docs. |

## Tier B: External pilot (stricter floor)

Before any real participant or identifiable household data, treat these as **hard gates** (from checklist stop-ship triggers + pilot gates):

| Item | Source in checklist |
| --- | --- |
| Pilot consent checklist completed | Pilot gates |
| Pilot operator checklist completed | Pilot gates |
| Pilot incident playbook with named contacts | Pilot gates |
| Pilot participant notice approved | Pilot gates |
| No publication of exact parcel-linked contributed data | Pilot gates + stop-ship |
| Sharing enabled only with matching notice | Stop-ship |
| Revocation reliably stops future sharing | Stop-ship (if sharing is on) |
| Parcel-linked operator access meaningfully tracked | Stop-ship |
| Claims / safety language reviewed for participant-facing materials | Governance + release copy gates |

## Explicit non-requirement

Tier A does **not** require closing revocation, consent backend, or full sharing UI — until you move to Tier B or claim product behaviors that depend on them.
