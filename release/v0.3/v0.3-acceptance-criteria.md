# v0.3 Acceptance Criteria

## Purpose

Define concrete, testable acceptance criteria that must pass before promoting
from v0.2 to v0.3.

## Acceptance tests

### AC-1: Flood packet normalization

**Given** a valid flood-node serial packet with
`schema_id: "oesis.flood-node.v1"` and `install_role: "low_point_primary"`

**When** the packet is submitted to `POST /v1/ingest/node-packets`

**Then** the ingest service normalizes it into a canonical observation with
flood-specific fields preserved (water_depth_cm, rise_rate_cm_per_hr,
calibration_state) and returns success.

### AC-2: Three-node registry binding

**Given** a parcel with `parcel_id: "parcel_test_v03"`

**When** `bench-air-01` (indoor), `mast-lite-01` (sheltered outdoor), and
`flood-01` (low-point) are all registered to that parcel

**Then** `GET /v1/parcels/parcel_test_v03/registry` returns all three nodes
with correct `node_class` values (`indoor_air`, `outdoor_reference`,
`low_point_flood`) and `enabled: true`.

### AC-3: Flood condition in parcel-state

**Given** parcel `parcel_test_v03` has a fresh flood-node observation with
`water_depth_cm: 8.3` and `rise_rate_cm_per_hr: 2.1`

**When** `GET /v1/parcels/parcel_test_v03/state` is called

**Then** the parcel-state response includes flood-relevant condition
information derived from the flood observation, with appropriate evidence_mode
and provenance entries.

### AC-4: Parcel view flood attribution

**Given** the parcel-state from AC-3

**When** the parcel view is rendered

**Then** the view distinguishes flood evidence from air quality evidence and
attributes flood observations to the low-point node.

### AC-5: Flood-node independent bring-up

**Given** the flood-node build guide and operator runbook

**When** a second operator follows the build guide

**Then** they produce a functioning flood-node that emits valid serial JSON
packets matching the serial-json-contract, including distance sensor readings
and derived water depth.

### AC-6: v0.2 regression

**Given** the v0.3 codebase

**When** `make oesis-validate`, `make oesis-check`, `make oesis-http-check`,
and `make oesis-v02-accept` are run

**Then** all v0.1 and v0.2 acceptance tests continue to pass.

## Contract boundary changes (v0.2 → v0.3)

- Ingest must accept `oesis.flood-node.v1` schema lineage
- Node registry must support `node_class: "low_point_flood"`
- Parcel-state must include flood-derived condition fields (additive, no
  breaking changes)
- Normalized observation model must accommodate flood sensor fields

## Acceptance command (proposed)

```
make oesis-v03-accept
```
