# v0.2 Acceptance Criteria

## Purpose

Define concrete, testable acceptance criteria that must pass before promoting
from v0.1 to v0.2.

## Promotion bar

Per `architecture/current/pre-1.0-version-progression.md`, promotion requires:

1. Explicit architecture scope — defined in `v0.2-scope-matrix.md`
2. Explicit contract and runtime boundary changes — documented below
3. Explicit acceptance commands or check updates — defined below
4. Explicit implementation-status evidence — tracked in `v0.2-implementation-status.md`

## Acceptance tests

### AC-1: Mast-lite packet normalization

**Given** a valid mast-lite serial packet with `schema_id: "oesis.bench-air.v1"`
and `location_mode: "sheltered"` and `install_role: "outdoor_reference"`

**When** the packet is submitted to `POST /v1/ingest/node-packets`

**Then** the ingest service normalizes it into a canonical observation with
outdoor metadata preserved (location_mode, install_role) and returns success.

### AC-2: Two-node registry binding

**Given** a parcel with `parcel_id: "parcel_test_v02"`

**When** both `bench-air-01` (indoor) and `mast-lite-01` (sheltered outdoor)
are registered to that parcel

**Then** `GET /v1/parcels/parcel_test_v02/registry` returns both nodes with
correct `node_class`, `location_mode`, and `enabled: true`.

### AC-3: Two-source inference

**Given** parcel `parcel_test_v02` has:

- a fresh indoor observation from `bench-air-01`
- a fresh sheltered outdoor observation from `mast-lite-01`
- a public context source

**When** `GET /v1/parcels/parcel_test_v02/state` is called

**Then** the parcel-state response:

- includes `evidence_mode` reflecting multi-source input (e.g., `local_plus_public`)
- includes provenance entries for both indoor and outdoor local sources
- confidence reflects having two local evidence streams

### AC-4: Parcel view source attribution

**Given** the parcel-state from AC-3

**When** the parcel view is rendered

**Then** the view clearly attributes which evidence came from the indoor node
vs the sheltered outdoor node, and the evidence summary distinguishes the two
source classes.

### AC-5: Mast-lite independent bring-up

**Given** the mast-lite build guide and operator runbook

**When** a second operator (not the original author) follows the build guide

**Then** they produce a functioning mast-lite node that emits valid serial JSON
packets matching the serial-json-contract.

### AC-6: v0.1 regression

**Given** the v0.2 codebase

**When** `make oesis-validate`, `make oesis-check`, and `make oesis-http-check`
are run

**Then** all v0.1 acceptance tests continue to pass. v0.2 must not break the
v0.1 baseline.

## Contract boundary changes (v0.1 → v0.2)

- Node registry must support `node_class: "outdoor_reference"` alongside
  `"indoor_air"`
- Parcel-state provenance must distinguish indoor and outdoor local sources
- No schema breaking changes — v0.2 is additive

## Acceptance command (proposed)

```
make oesis-v02-accept
```

This should run AC-1 through AC-6 and report pass/fail.
