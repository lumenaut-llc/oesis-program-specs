# v1.0 scope matrix

## Purpose

Map the **v1.0 release surface** honestly against PRD requirements and program-phase expectations from `architecture/system/product-requirements-phase-1.md` and `program/operating-packet/09-phasing-v0.1-v1.0-v1.5.md`, so reviewers can tell what is in the v1.0 scope, what is deferred, and what was never intended for this phase.

This matrix complements `v1.0-scope.md` (single-source release boundary) with a surface-by-surface breakdown. Use it alongside the implementation-status matrix to separate scope decisions from implementation status.

## Sign-off sentence (working definition)

**v1.0** means: the first materially broader fielded parcel-intelligence lane beyond the narrow v0.1 reference slice, adding mast-lite as a sheltered outdoor node, stronger trust scoring, deployment-quality flags, node registry binding, installation metadata, evidence summaries, basic sharing settings, and consent enforcement -- without claiming full consumer UX, push alerting, intervention logic, or complete route/block resilience.

## Explicit non-goals for v1.0

The following are intentionally outside v1.0 scope. They belong to later capability stages (v1.5, v2, or beyond) as defined in the phase roadmap.

- Full consumer app with push alerting, event timeline, and card-based UX
- Bounded controls, intervention ranking, or house-state response curves
- Full route/block resilience engine or neighborhood planning
- Public parcel-resolution shared map
- Thermal sensing as a general-purpose lane
- Weather-PM-mast as a default kit requirement
- Full controls-compatibility inventory
- Automated adaptation or recommendation execution
- Full revocation guarantees as a mature product (must be functional, not necessarily feature-complete)
- Flood as a central default hardware lane (remains optional per-parcel)

## Hazard coverage honesty (v1.0)

| Hazard | v1.0 honesty bar |
| --- | --- |
| Smoke / indoor air burden | Primary fit: bench-air indoor + mast-lite sheltered outdoor + public smoke feeds. Strongest evidence path. |
| Heat burden | Supported via temperature signals from both nodes + weather context. Not whole-home thermal mapping. |
| Flood / runoff | Optional where flood-node is installed and low-point sensing is operationally meaningful. Not a default claim. |
| Freeze / cold | Partial via temperature signals; no dedicated freeze instrumentation in v1.0 kit. |
| Outage / shelter | Inference may reference shelter/egress states; no backup-power sensing in v1.0 kit. |
