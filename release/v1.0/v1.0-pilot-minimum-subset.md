# v1.0 internal reference vs external pilot -- minimum subset

## Purpose

Map `v1.0-launch-checklist.md` to a **minimum subset** for two tiers:

1. **Internal reference v1.0** -- engineers and internal reviewers validate the widened parcel-kit slice (indoor plus sheltered outdoor, trust scoring, node registry); no external participant data.
2. **External pilot** -- any real household or participant data; stop-ship rules apply.

This document follows the pattern established by `../v.0.1/v0.1-pilot-minimum-subset.md` and raises the floor to match the v1.0 scope defined in `v1.0-scope-matrix.md` and `program/operating-packet/09-phasing-v0.1-v1.0-v1.5.md` (v1.0 section).

## Tier A: Internal reference v1.0 (minimum to claim widened slice)

Enough to demonstrate the first integrated parcel kit and trust-scored parcel state internally.

| Launch checklist area | Minimum items | Rationale |
| --- | --- | --- |
| Governance and legal | Data classes fixed for v1.0 contract lane (`make oesis-v10-validate`); privacy docs aligned with v1.0 scope; asset-class licensing matrix current. | v1.0 schemas must validate cleanly; legal posture must match the wider scope claim. |
| Product and UX | Provenance and freshness visible in reference `parcel_view`; node registry binds bench-air + mast-lite to one parcel; trust scoring penalties reflected in parcel state output. | These are the core v1.0 additions over v0.1: multi-node, trust-scored, wider evidence base. |
| Data and API | v1.0 contract schemas validate; reference pipeline runs with mast-lite lineage input; export bundle produces valid output. | The widened slice must be demonstrable end-to-end, not just documented. |
| Security and operator | Production-like demos use fixture data; operator access logging covers admin actions; operator runbooks current for bench-air and mast-lite. | No external data means fixture-based demos remain appropriate. |
| Pilot | **None required.** | No external participants at Tier A. |
| Hardware readiness | bench-air-node repeatable bring-up; mast-lite sheltered outdoor bring-up (build guide level, not field-hardened); integrated parcel-kit BOM understandable by non-author. | Tier A allows bench-grade and sheltered bring-up without requiring full field hardening. |
| Release copy | Public copy avoids directive safety language; stays within publication scope; README and NOTICE point to v1.0 summary. | Any published material must reflect the wider scope honestly. |

## Explicit non-requirements: Tier A

Tier A does **not** require:

- Closing revocation, consent backend, or sharing enforcement -- until you move to Tier B or claim product behaviors that depend on them.
- Continuous Wi-Fi transport or live public feeds -- serial extract and fixtures remain acceptable.
- Mast-lite field hardening -- sheltered bench-grade bring-up is sufficient.
- Named BOM vendor/SKU decisions -- procurement checklist with "bring your own" rules is acceptable.
- Evidence summaries in a dedicated UI -- reference JSON output is sufficient.
- Append-only history store -- snapshot-oriented reference path is acceptable.
- Installation metadata as guided product -- fixture-based context is sufficient.

## Explicit non-requirements: Tier B

Even at Tier B, the following are **not** v1.0 requirements:

- Full consumer app with push alerting, event timeline, or card-based UX.
- Full recommendation engine beyond reason strings and next-step flavor.
- Flood-node as a default kit requirement (only if parcel conditions warrant).
- Weather-pm-mast or thermal-pod in the pilot kit.
- Public parcel-resolution shared map.
- Bounded controls, intervention ranking, or house-state response curves (v1.5 scope).
- Full controls-compatibility inventory (v2.5 scope).

## Relationship to v0.1 baseline

v1.0 Tier A raises the v0.1 Tier A floor by adding:

- Multi-node parcel (bench-air + mast-lite bound through registry)
- Trust scoring penalties in parcel state output
- v1.0 contract lane validation alongside frozen v0.1 baseline

v1.0 Tier B raises the v0.1 Tier B floor by adding:

- Ingest authorization for live nodes (was deferred in v0.1)
- Wi-Fi/network transport with TLS (was deferred in v0.1)
- Live public feeds with fallback (was deferred in v0.1)
- Mast-lite field hardening as a hard gate
- Installation metadata capture
- Deployment-quality flags
- Named BOM vendor decisions for the Tier 2 kit

All v0.1 Tier B gates (consent checklist, operator checklist, incident playbook, participant notice, sharing/revocation enforcement, operator access tracking, claims/safety review) carry forward unchanged.
