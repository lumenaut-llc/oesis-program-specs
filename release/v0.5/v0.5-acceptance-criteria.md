# v0.5 Acceptance Criteria

## Purpose

Define concrete, testable acceptance criteria that must pass before promoting
from v0.4 to v0.5.

## Acceptance tests

### AC-1: Consent gates sharing

**Given** a parcel with sharing_scope `neighborhood_aggregate` and a valid
consent record granting `neighborhood_pm25` sharing

**When** a sharing query requests neighborhood PM2.5 data for that parcel

**Then** the query succeeds and returns coarsened data matching the consent
scope.

### AC-2: Missing consent blocks sharing

**Given** a parcel with no consent record for `network_assist` scope

**When** a sharing query requests network-assist signals for that parcel

**Then** the query returns empty — no data is shared without matching consent.

### AC-3: Revocation stops future sharing

**Given** a parcel with an active consent record for `neighborhood_pm25`

**When** the consent record's `revoked_at` is set (revocation event)

**Then** subsequent sharing queries for `neighborhood_pm25` from that parcel
return empty. Previously shared data is not deleted (mark-not-delete), but
no new data is emitted.

### AC-4: Retention cleanup executes and logs

**Given** a configured retention schedule (e.g., 90-day rolling window)

**When** the retention cleanup process runs

**Then** observations older than the retention window are removed, and an
auditable cleanup report is produced matching
`retention-cleanup-report.schema.json`.

### AC-5: Export bundle validates

**Given** a parcel with observations, consent records, and sharing settings

**When** an export bundle is generated for that parcel

**Then** the output validates against `export-bundle.schema.json` and includes
all parcel-linked data classes.

### AC-6: Operator access logging complete

**Given** an operator performs the following admin actions:
1. View parcel state
2. Update sharing settings
3. Process a rights request
4. Run retention cleanup

**When** the access log is reviewed

**Then** all four actions appear with timestamps, operator identity, and
action type.

### AC-7: Sharing settings configurable

**Given** a parcel with `parcel_id: "parcel_test_v05"`

**When** an operator sets sharing preferences via the sharing settings surface
(setting `network_assist: true`, `neighborhood_aggregate: false`)

**Then** the sharing store reflects the updated preferences, and subsequent
sharing queries respect them.

### AC-8: Custody tier enforcement

**Given** two consent records for the same parcel:
- one with `custody_tier: "internal_only"` for network-assist
- one with `custody_tier: "operator_visible"` for neighborhood aggregate

**When** sharing queries execute

**Then** network-assist signals are only available to internal model processes;
neighborhood aggregate signals are available to operator-visible surfaces.

### AC-9: End-to-end lifecycle

**Given** a fresh parcel with no governance state

**When** the following lifecycle is exercised:
1. Operator configures sharing settings
2. Consent record created for neighborhood_pm25
3. Sharing query succeeds
4. Consent revoked
5. Sharing query returns empty
6. Export bundle generated (includes revoked consent record)
7. Retention cleanup runs

**Then** each step produces the expected result, and the full lifecycle is
traceable through access logs.

### AC-10: v0.4 regression

**Given** the v0.5 codebase

**When** all prior acceptance suites run

**Then** all prior acceptance tests continue to pass.

## Contract boundary changes (v0.4 → v0.5)

- Consent-store becomes an operational runtime surface (not just schema)
- Sharing-store becomes an operational runtime surface
- Sharing queries enforce consent, revocation, and custody tier at query time
- Retention cleanup produces auditable reports
- Export bundle produces complete, validated output
- Operator access log covers all parcel-linked admin actions
