# v1.5 scope

## Purpose

Define what a bridge-stage `v1.5` packet would claim across architecture,
contracts, software, and hardware without overstating public-release maturity.

## Core claim

`v1.5` proves the minimum bridge from parcel sensing into response reasoning.

It should let the project claim one honest loop of:

`hazard -> house state -> action -> measured outcome`

with evidence-quality limits carried through to the result.

## In scope

- indoor-response measurement
- mains and backup-power posture
- read-side equipment-state capture where relevant
- action logging
- verification or outcome logging
- building and site metadata needed for response interpretation

## Out of scope

- full public consumer release claims
- broad automation claims
- full controls-compatibility inventory
- bounded-controls execution as a general product guarantee
- route and community resilience surfaces

## Packet honesty rule

If this lane is used for release framing, it should be explicit that:

- `v1.5` is primarily a capability-stage bridge
- some artifacts may still be draft or planned
- the packet is not proof that a mature adaptation engine already exists
