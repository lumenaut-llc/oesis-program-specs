# v1.0 Parcel Kit Architecture

## Purpose

Define the hardware and software architecture for the v1.0 parcel kit: the set of sensor nodes, runtime lanes, and inference capabilities that constitute the first materially broad environmental assessment surface for a residential parcel.

## Kit composition

The v1.0 parcel kit centers on a two-node core with an optional third node:

| Node | Role | Deployment | Schema |
|------|------|-----------|--------|
| **bench-air** | Indoor air quality baseline | Indoor, USB bench power | `oesis.bench-air.v1` |
| **mast-lite** | Sheltered outdoor conditions | Sheltered outdoor, USB power routed from indoor | `oesis.bench-air.v1` (shared lineage) |
| **flood-node** (optional) | Low-point water depth | Outdoor exposed, USB power | `oesis.flood-node.v1` |

### Shared lineage

Mast-lite uses the same `oesis.bench-air.v1` packet schema as bench-air. The distinguishing metadata is `location_mode: "sheltered"` in the packet, which the normalizer preserves and inference uses for evidence source classification. This shared lineage means:

- No separate normalizer is needed for mast-lite
- The same firmware scaffold works for both indoor and sheltered outdoor nodes
- The runtime differentiates nodes by `node_id` and `location_mode`, not by schema version

## Trust scoring

The v1.0 inference pipeline includes a 5-factor trust scoring model that quantifies the reliability of the current assessment:

| Factor | Weight | What it measures |
|--------|--------|-----------------|
| freshness | 0.30 | Time since last observation |
| node_health | 0.25 | Uptime, read failures, connectivity |
| calibration_state | 0.20 | Provisional, verified, or factory calibration |
| install_quality | 0.10 | Siting and enclosure discipline |
| source_diversity | 0.15 | Number and variety of contributing nodes |

Trust scoring feeds into the parcel view as `overall_band` (high, medium, low, degraded) and `overall_score` (0.0-1.0). A two-node kit with both nodes healthy typically scores in the "high" band. A single-node kit with stale data may degrade to "medium" or "low".

## Power architecture

All v1.0 nodes use USB 5V power:

- **Indoor (bench-air):** standard USB wall adapter, no special protection
- **Outdoor (mast-lite, flood-node):** USB power routed from indoor source through weatherproof cable path with drip loop, IP67 cable gland, and polyfuse protection

See [`parcel-kit/power-source-guide.md`](https://github.com/lumenaut-llc/oesis-hardware/blob/main/parcel-kit/power-source-guide.md) for power budgets, protection requirements, and open items.

## Deployment maturity

"Deployment maturity v1.0" means the reference software stack is complete and acceptance-tested, but hardware field hardening is not independently verified:

- **Software:** all 6 acceptance commands pass (v0.1 through v1.0); trust scoring, governance, multi-node composition implemented
- **Hardware:** bench-air is repeatable; mast-lite has a build guide but no independent field reproduction; flood-node has a build guide with provisional calibration only
- **Field readiness:** requires the shared field-hardening checklist to be satisfied (see [`parcel-kit/field-hardening-checklist.md`](https://github.com/lumenaut-llc/oesis-hardware/blob/main/parcel-kit/field-hardening-checklist.md))

## What v1.0 Tier A covers

Tier A (internal reference) proves the software stack with fixture data:

- Full pipeline for all three node types (bench-air, mast-lite, flood)
- Trust scoring computation and validation
- Governance enforcement (consent, revocation, retention, export)
- Multi-node evidence composition with calibration weighting
- Contrastive explanations and divergence analysis
- House state, intervention tracking, and verification outcomes

## What v1.0 Tier A does NOT cover (Tier B gaps)

- Ingest authorization for live nodes
- Wi-Fi or network transport with TLS
- Live public weather and smoke feeds
- Installation metadata guided input (uses fixtures)
- Sharing settings product UI
- Named BOM vendor decisions
- Mast-lite independent field reproduction
- Flood-node field validation beyond provisional calibration

These are tracked in `../../release/v1.0/v1.0-gap-register.md`.
