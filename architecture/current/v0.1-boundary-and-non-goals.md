# v0.1 Boundary and Non-Goals

## Purpose

Define what v0.1 IS and what it IS NOT. This prevents scope creep in the baseline lane and clarifies what later lanes (v0.2-v1.0) add.

## What v0.1 IS

v0.1 is the narrowest credible end-to-end path: one parcel, one sensor node, one pipeline, one parcel view.

- **One parcel** -- a single `parcel_id` with fixed context loaded from fixtures
- **One bench-air node** -- `oesis.bench-air.v1` schema, indoor USB-powered, serial JSON output
- **One pipeline path** -- ingest (normalize) -> inference (assess hazards) -> parcel platform (format view)
- **One parcel view** -- dwelling-facing status surface with condition estimates, confidence, evidence mode, freshness, provenance
- **Private-by-default** -- all data stays parcel-local; no sharing, no consent, no aggregation
- **Fixture-based context** -- public weather and smoke data loaded from checked-in example JSON, not live feeds
- **Snapshot-oriented** -- each pipeline run produces a point-in-time assessment; no persisted history

### Acceptance evidence

`make oesis-accept` proves v0.1 is working: schema validation, pipeline execution, output structure verification.

## What v0.1 IS NOT

v0.1 explicitly excludes the following capabilities. Each is addressed by a later lane:

| Capability | NOT in v0.1 | Added by |
|-----------|-------------|----------|
| Multiple nodes per parcel | v0.1 binds one node | v0.2 (2 nodes), v0.3 (3 nodes), v0.4 (registry lifecycle) |
| Outdoor nodes | v0.1 is indoor bench only | v0.2 (mast-lite sheltered outdoor) |
| Flood sensing | No flood observation family | v0.3 (flood-node) |
| Node registry lifecycle | No registration, validation, or filtering | v0.4 (registry management) |
| Governance and consent | No sharing, no consent, no revocation | v0.5 (full governance enforcement) |
| Trust scoring | No trust computation | v1.0 (5-factor model) |
| Contrastive explanations | No counterfactual reasoning | v1.0 (public-only counterfactual) |
| Divergence analysis | No local vs public comparison | v1.0 (disagreement detection) |
| House state influence | No interior condition input | v1.0 (house state, capability, intervention) |
| Shared intelligence | No neighborhood aggregation | v0.5 (consent-gated shared map) |
| Adaptation / controls | No equipment state or control surfaces | v1.0 (equipment state bridge) |
| Live public feeds | Fixture data only | Tier B (deferred) |
| Ingest authorization | Any client can post | Tier B (deferred) |
| Network transport | Serial only | Tier B (deferred) |
| Persistent history | Snapshot only | Deferred |
| Field-hardened power / enclosure | USB-only bench power; no IP rating | v0.2+ (per `../system/deployment-maturity-ladder.md` Deployment-class standards) |
| Battery / solar / mains-outdoor power | Not needed at indoor class | v0.2+ for sheltered-class nodes; v0.3+ for outdoor-class nodes |
| Characterized reference instruments | Not required at `deployment maturity v0.1` | v0.2 promotion bar per `../system/calibration-program.md` §G |
| Calibration-dataset admissibility | v0.1 readings are not admissible to coefficient fitting per calibration-program §C | v0.2+ (once §F / §G compliance is met) |

## Why this boundary matters

The v0.1 boundary prevents two failure modes:

1. **Overclaiming** -- describing capabilities that depend on later lanes as if they exist in the baseline
2. **Under-building** -- skipping the narrow path to chase broader features without a working foundation

Every lane from v0.2 through v1.0 layers onto v0.1 using the asset overlay mechanism (`oesis/assets/<lane>/`). Files not overridden in a later lane fall back to v0.1. This means v0.1 must remain stable and correct as the foundation for all later work.
