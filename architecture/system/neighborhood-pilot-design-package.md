# Neighborhood Pilot Design Package

## Purpose

Provide a practical structure for running an early block or neighborhood pilot that tests both product usefulness and the network-effect hypothesis.

## Pilot objective

Demonstrate that a small cluster of participating homes can produce:

- meaningful private home value
- improved shared local awareness
- acceptable privacy and governance experience

## Pilot scope

Recommended initial scale:

- one block, one short corridor, or one tightly bounded neighborhood segment

Recommended participant range:

- small enough to govern carefully
- large enough to expose real local variation

## Pilot design goals

- prove home-only value
- test whether nearby participation improves interpretation
- validate setup and context capture burden
- test whether shared intelligence feels helpful rather than invasive
- gather evidence for next-phase product and governance decisions

## Recommended hazard focus

Best pilot hazards:

- smoke and indoor air burden
- heat burden
- runoff and flooding where relevant
- outage readiness if likely in the pilot area

These are strong because they combine:

- practical parcel operator value
- visible local variation
- plausible neighborhood network lift

## Suggested pilot cohorts

### Cohort A: Home-only baseline

- participating homes receive private outputs only
- used to establish solo-home value

### Cohort B: Shared block intelligence

- participating homes receive private plus shared block outputs
- used to test network-effect lift

The pilot may compare these cohorts directly or phase them sequentially.

## Pilot setup components

### 1. Participant onboarding

- explain product purpose
- explain private versus shared outputs
- collect context fields
- explain consent and revocation

### 2. Sensor deployment

- install sensors with documented placement
- record installation mode and representativeness
- confirm connectivity and data quality

### 3. Baseline period

- run the system before acute events if possible
- establish everyday-value usage patterns

### 4. Event observation period

- observe one or more elevated conditions
- review home and block behavior during real events

### 5. Debrief

- interview users
- review logs and trust feedback
- assess privacy and governance fit

## Key pilot questions

### Home value

- did users find the product useful when no major event was happening?
- could users explain the current state and recommendations?

### Network effect

- did nearby participation reveal anything the private home view missed?
- did shared intelligence arrive earlier or feel more local than public context?

### Privacy and trust

- did participants understand what was shared?
- did the shared layer feel bounded and appropriate?

### Product clarity

- were recommendation cards understandable?
- did the map or shared layer create confusion?

## Minimum artifacts for each pilot

- participant notice
- consent and sharing explanation
- sensor placement record
- context-capture record
- event review template
- post-event interview guide
- privacy and trust survey

## Recommended instrumentation

- observation logs
- state-transition logs
- recommendation-view logs
- alert logs
- sharing-setting changes
- sensor-health logs

## Event review structure

For each meaningful event, review:

- what conditions occurred
- what the private home view showed
- what the shared layer showed
- what public context showed
- where the system helped
- where the system confused or overreached

## Success patterns

A strong pilot likely shows:

- users get value even on non-disaster days
- at least some events show clear block-level lift
- participants can explain the privacy posture
- the shared layer surfaces local variation public feeds did not make obvious

## Failure patterns

A weak pilot likely shows:

- users only care during rare crisis periods
- shared intelligence adds little beyond public feeds
- participants feel uncertain about what is shared
- sensor placement quality makes outputs hard to trust

## Follow-on decisions enabled by the pilot

- whether the block model is compelling enough to deepen
- which hazards deserve deeper investment
- whether operator roles are necessary
- whether the map or card view should lead
- which context fields are worth keeping in setup

## Open questions

- What is the smallest participant cluster that can still show a real network effect?
- Which pilot environments best expose local variation without requiring rare disaster timing?
- Should the first shared-layer pilot begin with maps, cards, or route outputs only?
